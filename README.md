# Лабораторная работа №2 — Детекция SMS-спама с помощью LLM

## Запуск

```bash
# 1. Поднять сервис
docker compose up --build

# 2. Установить зависимости (один раз)
pip install requests pandas scikit-learn

# 3. Запустить оценку всех техник
python scripts/evaluate.py

# Или выбрать конкретные
python scripts/evaluate.py -f zero_shot cot
```

Результаты сохраняются в `reports/details_<technique>.csv`.

---

## Постановка задачи

Цель работы — исследовать, как разные техники промптинга влияют на качество классификации SMS-сообщений на спам и ham. В качестве модели использовалась Qwen2.5:0.5B, запущенная локально через Ollama. Датасет — SMS Spam Collection (Kaggle), из которого случайно отобрано 100 spam и 100 ham сообщений (итого 200).

Модели давалось одно SMS, и она должна была вернуть JSON вида:
```json
{"reasoning": "...", "verdict": 1}
```
где `verdict=1` — спам, `verdict=0` — ham.

---

## Техники промптинга

Всего протестировано шесть техник:

- **zero_shot** — прямая инструкция без примеров и рассуждений
- **cot** — пошаговое рассуждение по пяти критериям (Chain-of-Thought)
- **few_shot** — пять размеченных примеров в промпте
- **cot_few_shot** — комбинация CoT и примеров
- **improved_cot** — трёхагентная структура: ConversationAgent, AdvertisingAgent, SpamAgent
- **improved_cot_few_shot** — трёхагентная структура с примерами

## Обсуждение результатов

### Few-shot примеры стабильно улучшают качество

Во всех трёх парах добавление примеров дало прирост F1:

- zero_shot → few_shot: **+0.058**
- cot → cot_few_shot: **+0.215**
- improved_cot → improved_cot_few_shot: **+0.023**

Это ожидаемо: маленькая модель лучше ориентируется, когда видит конкретные образцы правильного поведения, а не только инструкцию в абстрактной форме.

### CoT без примеров работает хуже нуля

Самый неожиданный результат — техника `cot` показала худшие метрики из всех, включая zero_shot. При F1=0.3066 и recall=0.21 модель явно начинает классифицировать большинство спама как ham. Судя по всему, жёсткая пошаговая инструкция заставляет модель 0.5B слишком формально следовать шагам, и она не находит явных индикаторов там, где они есть неявно. Помогло только добавление примеров — `cot_few_shot` сразу вышел на F1=0.52.

### Лучший результат — `improved_cot_few_shot`

Трёхагентная структура с примерами набрала accuracy=0.655 и F1=0.6188. Разделение ответственности на ConversationAgent, AdvertisingAgent и SpamAgent помогает модели не смешивать критерии и принимать более взвешенное итоговое решение.

### Проблемы

Сложные промпты (improved_cot и improved_cot_few_shot) привели к значительному числу ответов, которые нельзя было распарсить: 30 и 18 из 200 соответственно. Модель размером 0.5B не всегда способна поддерживать правильную JSON-структуру при генерации длинного reasoning. Для снижения числа ошибок формата в скрипте реализован механизм повторных запросов: если ответ не парсится, модель получает упрощённый промпт с требованием вернуть только JSON, до трёх попыток.

Отдельная проблема — ответы, которые технически являются валидным JSON, но содержат логическое противоречие между reasoning и verdict. Пример реального ответа модели:

```json
{"reasoning": "The message is clearly advertising a product or service, and the customer has given their name, so this message can be considered spam.", "verdict": 0}
```

Рассуждение указывает на спам, однако verdict=0. Это происходит потому, что финальный токен вердикта генерируется независимо от уже написанного текста — модель как бы «забывает» свой же вывод. Такие случаи в метриках парсинга не видны, но они напрямую снижают recall. Чтобы снизить число противоречий между reasoning и verdict, во все промпты добавлено явное правило: вердикт обязан следовать из рассуждения.

После доработки количество непарсируемых ответов сократилось до нуля — все шесть техник показали 200/200. Особенно заметно улучшение у `improved_cot` (было 170/200) и `improved_cot_few_shot` (было 182/200). При этом F1 у few_shot-техник вырос: `few_shot` +0.058, `cot_few_shot` +0.112. Техники без примеров (zero_shot, improved_cot) потеряли в F1 — добавленное правило про вердикт, судя по всему, сделало их более консервативными, что дополнительно снизило recall.

### Recall остаётся слабым местом

Precision у всех техник держится в диапазоне 0.57–0.69, тогда как recall значительно ниже — от 0.21 до 0.56. Модель систематически пропускает спам, принимая его за ham. Для реального применения в фильтрации это критично: большая часть нежелательных сообщений всё равно дошла бы до пользователя.

---

## Вывод

Для модели Qwen2.5:0.5B наилучшей техникой оказалась `improved_cot_few_shot` (F1=0.62). Ключевые наблюдения: few-shot примеры необходимы для стабильной работы маленьких моделей, CoT в одиночку может навредить, а сложные структурированные промпты повышают риск нарушения формата вывода. При переходе на модель большего размера абсолютные метрики должны вырасти, а проблемы с форматом — уйти.
